## Failure Modes

**Provenance laundering** is the most dangerous failure mode and the hardest to detect. An agent or human constructs a plausible-looking provenance chain — valid IRI structure, well-formed MDLD, passing SHACL validation — but the `prov:hadPrimarySource` URLs point to sources that have been manipulated, no longer exist, or never said what the statement claims. The vault's immune verification detects source disappearance and content change, but it cannot detect a source that was always false. This is the vault equivalent of fabricating citations in an academic paper. The structural validity of the provenance chain is not the same as the truthfulness of the grounded content. The mitigation is the corroboration requirement — no single-source statement can govern action — and the source credibility graph, which makes it harder to fabricate multiple independent institutional sources than a single one. But it cannot be eliminated entirely through automation. It is the residual human judgment problem.

**Shape capture** is a subtler failure mode. The S stroke gives an agent the ability to rewrite the rules of what counts as valid knowledge. A compromised or misaligned agent with S stroke authority could progressively loosen shapes — removing corroboration requirements, lowering source credibility thresholds, eliminating temporal bounds — until the vault accepts almost anything. Each individual S stroke looks like a reasonable optimisation; the cumulative effect is a vault that has silently abandoned its epistemological standards. The mitigation is the S stroke's provenance chain — every shape revision is a grounded entity with a rationale literal and a `prov:wasRevisionOf` chain — combined with a meta-shape that flags any revision that reduces minimum constraint counts. A shape revision that results in fewer constraints than its predecessor generates a violation on the meta-shape, requiring human approval before it can be committed. Tightening is automatic; loosening requires a human signature.

**Source drift** is the slow failure mode. A URL that was valid and correct when grounded gradually becomes incorrect as the underlying content changes — a journal changes its editorial standards, a government dataset is revised without a new URL, a Wikipedia article (if used as a source, which it should not be) is edited. The immune verification cycle catches binary changes — URL gone, content hash changed — but cannot detect subtle semantic drift where the URL still resolves and the content is recognisably the same document but the specific claim the statement relied on has been quietly amended. Mitigation is the snapshot archive requirement: for any statement used in `vault preconditions`, the source content at grounding time should be archived via the Wayback Machine or a content-addressed store. The `prov:hadPrimarySource` URL is supplemented by a `p:archivedSource` pointing to a Wayback Machine permalink with a specific timestamp. Semantic drift cannot invalidate an archived snapshot.

**Context window hallucination during construction** is the failure mode specific to LLM-operated vaults. When a large vault is being constructed and the LLM receives a compressed graph summary rather than the full graph, it may make grounding decisions based on an incomplete picture of what is already in the vault — missing contradictions, failing to cross-reference related entities, duplicating statements with slight variations. The mitigation is architectural: the agent should never make grounding decisions without running `vault contradictions` and `vault similar` first, and these calls are deterministic regardless of context window size. The SPARQL engine sees the full graph; the LLM only needs to see the results. The vault CLI must make it impossible to run `vault action complete` without a preceding `vault contradictions` call in the same session.

**Federation poisoning** happens when a compromised vault is federated into a healthy one. The `vault federate` command brings in entities with `prov:wasDerivedFrom` links to the source vault, but if the source vault's grounding chain has been corrupted, the corruption travels through the federation link. The mitigation is the federation authority shape — only VERIFIED source vaults can be federated, and VERIFIED status requires the source vault's own `vault immune integrity` to pass and its shape lineage to be publicly auditable. Federating a private vault with no public git history into a public vault should require explicit human approval and creates an audit event.

**Re-identification through provenance** is a privacy failure mode that becomes relevant when vaults contain personal or sensitive data. A provenance chain that traces a grounded statement back to a specific person's query at a specific time is a deanonymisation vector. If a medical vault records that a particular diagnosis was grounded through an activity associated with a specific patient record, the provenance chain is a liability. Mitigation is provenance compartmentalisation: sensitive activities are recorded in a separate private vault that is never federated, and the public vault contains only the grounded statement with a generic institutional source — the hospital, not the patient record. The `p:privacyScope` property on activity entities, combined with a shape that prevents private-scope activities from being federated, enforces this structurally.

## Cryptographic Strengthening

Git already uses SHA-1 (transitioning to SHA-256) for object integrity — every commit hash is a cryptographic fingerprint of its content and its parent chain. This is necessary but not sufficient. What git does not provide by default is **identity binding** — the commit author field is a self-declared string, not a cryptographically verified identity. Anyone can commit as anyone. The vault needs three additional cryptographic layers beyond what git provides out of the box.

**Signed commits via GPG or SSH keys** bind every commit to a verified key pair. Every agent, human contributor, or automated pipeline that writes to the vault has a key. The vault CLI enforces signing at write time — `vault write` calls `git commit -S` automatically. A commit without a valid signature is rejected by the repository's pre-receive hook. This means the author field in `git log` is not just a string — it is a claim backed by a cryptographic signature that can be verified against a published public key. The `vault authority` command's check against the `vx:agent:orchestrator` IRI becomes meaningful only when that IRI is linked to a verified public key. Without signing, the authority check is advisory; with signing, it is enforced.

**Content-addressed source archiving** solves the source drift and provenance laundering problems simultaneously. When the D2 grounding stroke commits a new `prov:hadPrimarySource` URL, the vault CLI fetches the source content, computes its SHA-256 hash, and stores both the URL and the hash. The `prov:hadPrimarySource` is supplemented by `p:sourceHash` containing the content fingerprint. The immune verification cycle then checks not just "does this URL still resolve?" but "does this URL still resolve to the same content?" Any content change, however subtle, changes the hash and triggers a trust flag. For truly critical statements, the source content is published in as a content-addressed file - with it's hash as the name/metadata — making the reference permanently verifiable regardless of whether the original server persists.

**DID-anchored vault identity** addresses the federation trust problem. A vault's identity — its claim to be the authoritative L0 vault, or the L3 provenance science vault, or a specific domain specialist vault — currently rests only on its git remote URL, which can be spoofed or transferred. Decentralised Identifiers (DIDs, a W3C standard) provide cryptographically anchored identity independent of any particular hosting provider. The vault's DID is generated at `vault init`, published to a DID registry (could be a blockchain anchor, could be a web DID on the organisation's own domain), and included in `vault/self.md`. Every federation request includes a DID proof — a cryptographic signature proving the offering vault controls the private key corresponding to its DID. A vault claiming to be the L0 foundation without a valid DID proof for that identity is rejected by the federation authority shape.

**Merkle-anchored snapshots** provide external tamper evidence beyond what git alone gives. Periodically — daily for active construction vaults, weekly for frozen vaults — the root commit hash of the vault is anchored into an external, independently verifiable record. Options range from a public blockchain (Ethereum, Bitcoin via OpenTimestamps) to a trusted timestamping authority (RFC 3161, which is legally recognised in many jurisdictions). The anchor creates a proof that the vault existed in a specific state at a specific time, witnessed by a system outside the vault's own control. A vault that claims it was built over six months but whose entire commit history was fabricated in a single session cannot produce valid RFC 3161 timestamps for each intermediate commit. The temporal anchoring makes retrospective fabrication computationally or legally infeasible.

**Threshold signature for S stroke** extends signing to the most consequential operation. Rather than requiring a single orchestrator key to authorise shape revisions, a threshold signature scheme (Shamir's Secret Sharing or a multi-signature protocol like FROST) requires that M of N designated key holders co-sign any shape revision before it is accepted. For a company vault, this might be two of three: the automated agent, the domain expert, and the data governance officer. For a public knowledge vault, it might be three of five community validators. The threshold can be set low for routine shape tightening and high for shape loosening or pivoting — exactly calibrated to the risk profile of each operation type. No single compromised key can steer the vortex.

## The Resulting Integrity Model

With these additions the vault's integrity properties become formally statable rather than aspirational:

A statement in the vault is **structurally valid** if it passes SHACL validation against the current shape set. It is **cryptographically attributed** if its commit carries a valid GPG or SSH signature linked to a known key. It is **temporally anchored** if its commit hash appears in an RFC 3161 timestamp or blockchain anchor. It is **content-verified** if its `p:sourceHash` matches the current content of its `prov:hadPrimarySource` URL. It is **identity-bound** if its source vault has a valid DID proof. And it is **corroborated** if at least two independent VERIFIED sources with institutional independence scores above threshold both support it.

A statement that satisfies all six properties is as trustworthy as the current state of the art in formal knowledge systems allows. It cannot be fabricated without breaking at least one of the six properties in a way that is detectable by automated verification. It cannot be silently modified without invalidating the git object hash chain. It cannot be backdated without forging the external timestamp anchor. It cannot be falsely attributed without stealing a private key. The residual risk — that the sources themselves are wrong, that the institutional independence scoring missed a hidden coordination, that the domain expert who signed the S stroke was mistaken — is the irreducible human judgment risk that no cryptographic system can eliminate. But it is a well-characterised, bounded, and auditable residual risk rather than the unbounded, opaque, and unauditable risk of current AI knowledge systems.

The vault is a git repository. Git is forty years of accumulated engineering wisdom about distributed, append-only, cryptographically integrity-checked content storage. The vault does not replace that foundation — it sits on top of it and adds the semantic layer that turns a repository of files into a repository of grounded, verifiable, action-governing knowledge. The ZIP is the unit of distribution. The git history is the unit of trust. The SHACL shape set is the unit of epistemology. And the cryptographic signatures are the unit of accountability.
